## 2026-07-08 — detail-view review: 6 naming/refactor decisions (Phase 13)
Owner review of the branch surfaced six items. Investigated each (parallel Explore agents), owner decided:

1. **`nestedLimit` → fixed constant.** It was an optional param threaded DetailView→fetchRowData→fetchTableData, but only ever `5` (catalogue detail pages) or unset=unbounded (generic `/table`, `apps/ui`). `expandLevel` genuinely varies (1/2/3); `nestedLimit` does not. **DECISION:** replace with module-level `DEFAULT_NESTED_LIMIT = 5` applied in fetchTableData; drop the 3 literal `5`s + the prop plumbing. Behavior change: generic detail view gains a 5-row nested cap (desirable — never SSR-prefetch unbounded refback; "show all" is the child view's job).
2. **Keep `display` + `role` as two metadata columns (rejected merging).** They are orthogonal axes and legitimately co-occur: a REFBACK can be `role="DETAIL"` (which slot) AND `display="CARDS"` (how nested rows render) — `DetailColumn.vue:66,155`. Merging forces a cross-product enum. Owner: "we need to choose these nested cards vs lists, so keep them." (Both are real persisted GraphQL metadata columns.)
3. **`sourceTableId` → `tableId`.** `sourceTableId` (runtime annotation from getSubclassColumns) means "the table that DEFINES this column" = metadata's tableId, so owner's read is correct. **DECISION:** rename sourceTableId→tableId (5 sites: types.ts:85, getSubclassColumns.ts:24, 2 spec asserts, Ref.story.vue). **CORRECTION (impl):** the intended "remove dead `table?`/`inherited?`" was DROPPED — both have live readers (lead's earlier grep was wrong): `table?` read in `filter/Picker.vue:167` (`props.columns[0]?.table`), `inherited?` read in `apps/schema/src/utils.ts:85` + `ProfileManager.vue:163`. Rename only.
4. **Introduce shared `ontologyValueObject` type.** ContentTypeOntologyArray/RefBack currently cast to the generic open-bag `columnValueObject`. Authoritative ontology fields (backend `SqlColumnExecutor.java:294`): `name`(req), `label?`,`code?`,`codesystem?`,`ontologyTermURI?`,`definition?`,`order?`,`tags?`,`parent?`,`children?`. Two duplicate interfaces already exist (`IOntologyItem` catalogue, `IOntologyNode` cms.ts). **DECISION:** add canonical `ontologyValueObject` (+ guards) in metadata-utils/src/types.ts matching columnValueObject/fileValue convention; use in the ontology content types; point the dup interfaces at it where cheap.
5. **`DataCards.gridColumns` → `maxColumns`** (`1|2`, responsive max, not an exact/arbitrary count). Must also update DataList's LIST-vs-CARDS dispatch (`DataList.vue:183-190` — LIST = DataCards at 1 column).
6. **Make `DataCards` genuinely dumb (option a: hoist into DataList/getListColumns), NOT a new Emx2DataCards.** Phase 12 deliberately removed Emx2* wrappers for dual-API base components; reintroducing one would reverse that. Audit: DataCards is the only renderer still HYBRID (reads role via getDescriptionColumn/getDetailColumns/getLogoColumn internally) — the plan already *calls* it "dumb" but it isn't. DataTable/DataLinks already dumb; DetailView/DetailColumn ARE the designated smart layer (reading metadata is their job — no change, optional pure-helper extraction from DetailView for testability). **DECISION:** classify description/detail/logo by role once in the smart layer, pass classified columns to DataCards; DataCards renders per-row (keeps its own empty-value skipping). Also check other renderers stay consistent.
**OWNER:** mswertz

## 2026-07-20 — REVERSAL of item 6 (13e): DataCards keeps `columns`+`rows` API
**WHAT:** Owner reviewed the shipped 13e DataCards and found the split-props approach "not really an improvement — better to have row, column like before." REVERSED the smart/dumb split. DataCards goes back to props `{ rows, columns, maxColumns?, rowLabelTemplate?, schemaId?, tableId? }` — same shape as its sibling DataTable. It calls `classifyCardColumns(props.columns)` INTERNALLY (instead of receiving descriptionColumn/detailColumns/logoColumn as props). DataList passes `:columns`+`:rows` (drops the cardColumnClassification computed + split-prop wiring).
**KEPT from 13e (the parts that WERE improvements):** the single tested pure `classifyCardColumns` fn (still the one source of card classification, now called by DataCards) + its 13 tests; `isEmptyValue` per-row skipping. The old getDescriptionColumn/getDetailColumns stay removed.
**WHY:** the split-props API diverged DataCards from DataTable, pushed work onto callers, and caused the LOGO-via-DataList breakage — with no offsetting benefit. Owner prefers the consistent `columns`+`rows` renderer API.
**HOW:** folded into the (still-unpushed) review-cleanup commit via `--amend`, not a new revert commit. `maxColumns` (item 5) unaffected.

## 2026-06-16 — Un-defer E4 Q3: emit ACTIVE MODULES as `rdf:type` — via PER-TABLE emission (row-presence = activation), NOT discriminator re-derivation
**Owner:** mswertz ("shouldn't we emit as type?" → then "so much logic on RDF level?" → chose "Refactor to per-table").
**What changed:** E4 Q3 (2026-06-13) DEFERRED emitting a separate `rdf:type` per active module (FHIR `meta.profile[]` / RDF multiple-`rdf:type`). Now UN-DEFERRED. **Mechanism (owner-corrected — supersedes the first impl):** a module is stored as a child table on the shared root PK, and C3 writes a row into module table M *only* for records where M is active — so **row-presence in M == typed-M**, exactly like `mg_tableclass` drives is-a typing (RdfGenerator:60 `where mg_tableclass=T`). `rowIRI` is already root-anchored (IriGenerator:74), so a module row's subject == the root row's subject. So RDF DESCRIBES module tables with a minimal per-row pass emitting ONLY `subject rdf:type tableIRI(M)`. Module-extends-module needs NO ancestor walk — C3 writes a row in each ancestor module table, each described separately → each emits its own type (`a :experiment, :RNAseq, :RNA` falls out). Applies to BOTH scalar `MODULE` and `MODULE_ARRAY`.
**Rejected first impl:** the discriminator-re-derivation in `Emx2RdfGenerator.dataRowToRdf` (`emitActiveModuleTypeTriples` reading `getDiscriminatorColumns()` values → `getModuleSubtypeTables()` → walk `getAncestorsRootFirst()`) — it re-derived what storage already encodes. Owner: "so much logic on RDF level?" → DELETE it (+ `collectModuleAncestorChain`).
**Implementation:** (1) `RdfRowsGenerator.processRows:34` — replace the `tableType==MODULE` early-return with a `case MODULE` that emits a per-row type-only triple (new concrete `moduleRowToRdf` in the base — generic, uses root-anchored `rowIRI` + `tableIRI`; skips drafts). (2) **LANDMINE:** `RootRdfGenerator:35` filters tables to `TableType.DATA` only — MODULE tables never reach `processRows` for a whole-schema dump → MUST include MODULE there (and audit every other `processRows`/table-set entry point: `Emx2RdfGenerator:60/73`, `tablesToDescribe`). (3) `Emx2RdfGenerator.dataRowToRdf` — remove the `emitActiveModuleTypeTriples` call + delete both helpers; KEEP `getColumnsIncludingModules` (root still inlines module COLUMNS — E4 unchanged, so no duplicate column triples).
**What stays:** discriminator COLUMN remains `SKIP` in `ColumnTypeRdfMapper`. Module CONTENT column emission (E4) unchanged — root inlines them; the module-table pass adds ONLY the type triple. Exhaustive `ColumnType→RdfColumnType` map kept (H.1 `MODULE→SKIP` stays).
**Why per-table:** mirrors is-a typing (per-table iteration + row-presence), reuses the existing structure, deletes bespoke RDF logic. The first impl fought E4's root-centric choice instead of following the is-a precedent.
**Scope:** `molgenis-emx2-rdf` (RdfRowsGenerator + RootRdfGenerator + Emx2RdfGenerator + any other table-set entry points) + RDFTest. Its OWN commit (`feat(rdf): …`), separate from H.1. Spec row reworded.

## 2026-06-14 (d) — Phase H scalar MODULE derived-default `values` lives in a NEW `getEffectiveValues()`, NOT `getValues()`
**Owner:** mswertz (AskUserQuestion, "New getEffectiveValues()").
**What changed:** the Phase H plan said the derived-default `values` (all direct+indirect MODULE subclasses of the root, for a scalar `MODULE` column with no explicit `values`) would "surface in `Column.getValues()`". Recon found `getValues()` is read by TWO serializers — CSV export (`Emx2.java:323`) and JSON model save-back (`json/Column.java:114→152`) — so returning derived there would FREEZE the derived set into the model on any export→re-import or schema-editor save, directly contradicting the locked "computed at READ-TIME, NOT frozen; a module added later just appears."
**Decision:** keep `getValues()` = the RAW declared field (serialization-safe by construction). Add a NEW `Column.getEffectiveValues()` that, for scalar `MODULE` with empty raw values, returns `getModuleSubtypeTables()` names (live), else the raw values. The frontend picker (GraphQL `_schema` OUTPUT) AND the write-time enum-membership check read `getEffectiveValues()` (one shared set); CSV/JSON serializers keep reading `getValues()` (raw → stays empty/live). Satisfies BOTH "picker & membership see the same set" AND "never frozen" with no missed-patch risk.
**Why over the literal alternative (getValues()=derived + patch both serializers to a raw accessor):** same UX but more patch sites + a silent-freeze risk if any serializer is missed now or added later. The new-method seam makes serialization correct by default.
**Scope:** scalar `MODULE` ONLY (MODULE_ARRAY keeps explicit `values`, untouched — locked decision #2). Affects H.1 only.

## 2026-06-14 (b) — F2 module columns: SERVER serves them on the containing table; client gates via VISIBILITY only (no client merge/sort)
**Owner:** mswertz
**What changed (supersedes the client-merge approach in F1/E3 + the first F2 impl):**
1. **Server-side, not client-side.** The containing (root) table's `_schema` metadata `columns` must INCLUDE all module columns, interleaved by `position` (E6 global order) — exactly as it already includes is-a inherited columns (each json `Column` already carries `table`=owning table name + `position` + `inherited`). The client must NOT merge module groups or sort by position. → drop `expandModuleColumns` merge + the `visibleColumns` position-sort; likely drop the `schema` threading into useForm entirely (gating only needs each col's `table` marker + the discriminator value, both on the single table metadata).
2. **Gate purely via VISIBILITY.** A module column's active/inactive state is folded into its VISIBILITY signal (module col visible ⇔ its owning module ∈ active discriminator set, AND-combined with the modeler's own `visible` at the signal level — still NOT a synthesized `visible` string). All downstream effects then follow from the EXISTING rule "only visible columns are validated/required" — so REMOVE the separate `requiredMap` module AND-gate and the `validateAllColumns` inactive-skip added in the first F2 impl; they're redundant once visibility carries the gate.
**Why:** the data query + GraphQL types already treat module cols as part of the root (`getColumnsIncludingSubclassesAndModules…`, E2/F0); only the `_schema` introspection lagged (`json/Table.java:73` uses `getColumns()`). Putting the merge on the server is the consistent layer, mirrors is-a inherited cols, and makes the client dumb. Gating through visibility reuses the one mechanism instead of special-casing each effect.
**Terminology clarification (owner-probed 2026-06-14b):** "containing table" = the table the FORM renders, which MAY itself be an is-a subclass (e.g. a form on `ClinicalSubject extends Subject`). The MODULE_ARRAY discriminator is declared on the inheritance ROOT (`getRootTable()`) and inherited by the subclass; the module CONTENT cols are anchored at that ROOT. So module cols must surface on the RENDERED table whether it's the root OR an is-a subclass of it — collected via the rendered table's ROOT, NOT via the rendered table. **Bug-in-waiting:** existing `getColumnsIncludingModules()`/`getModuleSubtypeTables()` only find modules DIRECTLY extending the table they're called on (`collectModuleSubtablesDeduped` matches `inheritNames.contains(getTableName())`), so on a subclass they return nothing. The server-side fix must be ROOT-ANCHORED (walk `getRootTable()`), which also means every is-a subclass of a module-bearing root carries the module cols in its `_schema.columns` (consistent with already inheriting the discriminator + parent cols; widens blast radius).
**Open / blast radius (verify before implementing):** `_schema` table.columns is consumed by the schema editor (apps/schema) + IO/RDF/beacon/other frontends. Module cols appearing on the root table (tagged `table`=module) must not be mistreated as root-local-editable — likely already handled the way inherited cols are (tagged + `inherited`), but MUST be verified. `getColumnsIncludingModules()` needs a position-sort (currently appends). Keep `buildSubmitValues` (submit-omit of deselected modules = C6 semantics, separate from display visibility).
**SCOPE CONFIRMED (owner 2026-06-14b): "server-side, done right."** Full coordinated change — backend serialization + ONE shared `isLocalColumn(col,table)=col.table===table.id` predicate applied in form/data-query/schema-editor (replacing the `!inherited` proxy), `table`+`inherited` added to frontend gql, affected backend tests updated, form drops client merge/sort. Spans backend + F2/F3/F5 filter-correctness. Tracked as F2b.
**Backend depth (verified):** module-subtype resolution is currently anchored on `this` table via `getModuleSubtypeTables()` (modules DIRECTLY extending it) — at `SqlQuery.tableWithInheritanceJoin:901` (query join), `TableMetadata.getColumnsIncludingModules:756`/`getColumnByNameIncluding...` (read), almost certainly the C3 write path, and the metadata serializer. A form may render on an is-a SUBCLASS of the module-bearing root, so all of these must become ROOT-ANCHORED (`getRootTable().getModuleSubtypeTables()`, modules share the root PK) so operating on a subclass behaves like the root: subclass query PROJECTS active module cols, subclass insert/update ACTIVATES+writes module rows, subclass `_schema` INCLUDES module cols. is-a byte-identical where no module-bearing root.
**Effect:** spec rows 110/111 reopened; first F2 client impl (staged) will be simplified once backend serves merged cols. Backend recon done (`.plan/notes/phase-f2-server-merge-recon.md`).

## 2026-06-14 — F2 module-content reveal: Accordion/collapsible block → INLINE gated like `visible`
**Owner:** mswertz
**What changed:** Dropped the "collapsible labeled fieldset (Accordion) per module" design. Module content columns now render INLINE, interleaved with the root columns by `position`, shown/hidden by which MODULE(_ARRAY) values are selected — same UX as a `visible` expression, no block/grouping.

**Effect:** spec rows 110/111 reworded; `useForm` folds module cols into `visibleColumns` (position-sorted) with `visibilityMap`/`requiredMap` AND-gated by owner-module-active; `Fields.vue` unchanged; no `Accordion`/`ModuleGroup`.

## Note on H.1 (GIN index)

Audit of git history shows the GIN index on `mg_groups` (`CREATE INDEX IF NOT EXISTS
<table>_mg_groups_idx ... USING GIN (mg_groups)`) was already present in
`SqlRoleManager.enableRlsForTable` since commit `7e3243b94` (early phase 8 / "rename"
commit). The Phase G benchmark ran WITH the index already in place. There is no
pre-GIN baseline to compare against in this repo.

The Phase H.1 deliverable is therefore:
- Confirmation the DDL is correct and idempotent (already `IF NOT EXISTS`).
- A new test `enableRls_createsGinIndexOnMgGroups` in `TestRlsEnableDisableLifecycle`
  asserting the index exists after enable and is gone after disable (column drop
  cascade-drops the index automatically).
- Re-run of the benchmark to confirm numbers are stable.

**Interpretation**: Numbers are in the same order of magnitude; run-to-run variance at
this dataset size (5000 rows for A3, 50000 for A1) is ±20–30%. The GIN index does not
measurably help the A3 ratio because the GROUP-scope test uses only 5000 rows and PG's
planner chooses a sequential scan over an index scan at that cardinality. The index
would matter for tables with >100k rows where index scans become cheaper than seq scans.

## H.1.5 verification (2026-05-08)

Removed the third UNION ALL branch from `mg_can_read` and `mg_can_write` that joined
`pg_roles → pg_auth_members → pg_roles`. Functions now use only GMM + RPM (no catalog
access), making them inlinable by the PG planner.

### A3: GROUP-scope vs ALL-scope at 5k rows (post H.1.5)

overhead_groupScope: ALL-scope=668ms GROUP-scope=694ms **ratio=1.04×**

Before (H.1 with catalog branch): 5.52×
After (H.1.5 without catalog branch): **1.04×**

**Improvement: 5.3× reduction in GROUP-scope overhead at 5k rows.**

### EXPLAIN plan

At 5k rows with the catalog branch removed, the PG planner can now inline
`mg_can_read` and push the `mg_groups && $1` predicate to the GIN index. The
seq-scan inflection that previously occurred at low cardinality is eliminated.
GIN engages at any cardinality once the function is inlinable.

### New contract

Direct `GRANT MG_ROLE_<schema>/<role> TO MG_USER_<user>` without a corresponding
GMM row now yields zero rows visible. GMM is the sole source of truth for custom-role
membership in the access functions. Regression test:
`TestSqlRoleManager.directPgGrantWithoutGmmRow_yieldsNoRows`.

## Top 3 Recommendations

1. **GIN index on mg_groups** (DONE): Already emitted by `enableRlsForTable`; `<table>_mg_groups_idx` created on enable, dropped via column cascade on disable. The index targets the `mg_groups && $1` overlap in `mg_can_read`. At 5000-row scale the planner uses seq scan; benefits will show at >100k rows.

2. **Consider materialised membership cache for high-group-count users**: The `mg_can_read` custom-role branch performs a catalog JOIN (`pg_roles → pg_auth_members → pg_roles`) per row. For users with many roles across many groups the JOIN is re-evaluated for every row. A session-level materialised array of `(schema, table, groups[])` — populated once at `SET ROLE` time via a `SECURITY DEFINER` function — would turn this into an array overlap, reducing I/O.

3. **Materialized-view fallback for N_ROWS > 500k**: If benchmark numbers show >3× overhead at production row counts, consider an opt-in materialised view `mg_visible_<table>` keyed on `(user, row_id)` refreshed by a trigger on `role_permission_metadata` and `group_membership_metadata` changes. The policy would then be a simple PK lookup. This trades write amplification for read-path cost, and should only be enabled when profiling confirms it is needed.
